C'est quoi HLS4ML ?

High Level Synthesis for Machine Learning — le pont entre Python et le FPGA

Normalement, programmer un FPGA demande d'écrire du VHDL ou Verilog — des langages hardware très bas niveau. HLS (High Level Synthesis) permet déjà d'écrire en C/C++ et de le convertir automatiquement en RTL. HLS4ML va encore plus loin : tu lui donnes un réseau de neurones Keras/PyTorch, il génère tout le code HLS tout seul.

🧠 Étape 1 — Python

Tu entraînes ton réseau normalement avec Keras en float32. Le modèle tourne sur CPU/GPU.

⚙️ Étape 2 — HLS4ML

HLS4ML convertit le modèle en code C++ HLS optimisé pour FPGA. Aucune connaissance VHDL requise.

⚡ Étape 3 — FPGA

Vivado/Vitis synthétise le C++ en bitstream. Inférence en quelques nanosecondes.

Origine : CERN / LHC

HLS4ML a été créé par la communauté de physique des hautes énergies du CERN. Les détecteurs du Large Hadron Collider produisent des milliards de collisions par seconde — il faut trier les événements intéressants en temps réel, avec des latences inférieures à la microseconde. Impossible sur CPU. Le FPGA + HLS4ML résout exactement ce problème.

Pourquoi le FPGA ne fait pas de virgule flottante ?

Un CPU peut faire des calculs float32 facilement car il a des unités de calcul flottant dédiées (FPU). Sur un FPGA, tout est câblé manuellement en portes logiques. Implémenter float32 sur FPGA consommerait des ressources énormes. On utilise donc des nombres en virgule fixe : ap_fixed<N,I>.

Le défi principal de cette Projektarbeit

Trouver la bonne précision ap_fixed pour que le modèle tienne dans les 220 DSPs du PYNQ-Z1, tout en maintenant une accuracy proche du modèle Keras original. Les Parts 1 à 3 échouent toutes. C'est seulement à partir de la Part 4 (QKeras) que ça rentre.

FPGA & PYNQ-Z1

Les ressources disponibles sur ton board — les limites à ne pas dépasser

Un FPGA est un circuit intégré composé de blocs logiques programmables. Il n'exécute pas d'instructions comme un CPU — il câble physiquement les opérations dans le silicium. Ce qui le rend ultra-rapide pour des calculs parallèles répétitifs comme l'inférence neurale.

PYNQ-Z1 vs PYNQ-Z2 — une subtilité importante

HLS4ML supporte officiellement le PYNQ-Z2, pas le PYNQ-Z1. Mais les deux utilisent le même chip Xilinx Zynq xc7z020clg400-1. Solution utilisée dans ta Projektarbeit : spécifier le part='xc7z020clg400-1' directement et ajouter manuellement les fichiers Board Definition du PYNQ-Z1 dans les frameworks.

⚠ Les DSPs sont la contrainte principale

Chaque multiplication dans un réseau dense nécessite des DSPs. Un réseau avec 480 multiplications en RF=1 a besoin de ~480 DSPs — plus du double de ce que le PYNQ-Z1 possède. C'est pour ça que les Parts 1, 2, 3 ne rentrent pas : il faut réduire les DSPs via le ReuseFactor ou la quantisation.

Comment HLS4ML utilise ces ressources

DSPs ← multiplications

Chaque neurone d'une couche Dense fait des sommes de produits (poids × activation). Chaque produit = 1 DSP en RF=1. Formule : DSPs = n_weights / RF

LUTs ← logique

Additions, fonctions d'activation (ReLU, Softmax), multiplexeurs, contrôle de pipeline. Les LUTs augmentent avec la complexité architecturale.

⑤ Pruning et Sparsité

Pruning = mettre délibérément des poids à zéro pendant l'entraînement. Si un poids est zéro, la multiplication correspondante est nulle → le DSP peut être ignoré. HLS4ML exploite cette sparsité quand le RF est un multiple du pattern de sparsité.

Magnitude Pruning avec TensorFlow Model Optimization Toolkit

Les plus petits poids en valeur absolue sont progressivement mis à zéro pendant l'entraînement. Part 3 : 50% de sparsité → ~2× moins de DSPs théoriquement. Part 4 : 75% de sparsité combiné avec QKeras → résultats bien meilleurs.

⑥ Latency et Initiation Interval (II)

Latency

Nombre de cycles d'horloge entre l'entrée d'un sample et la sortie du résultat. Sur PYNQ-Z1 à 200MHz : 50 cycles = 0.250 µs.

Initiation Interval (II)

Combien de cycles avant de pouvoir accepter un nouveau sample. Si II=1 (pipelining complet), le débit est maximal. Si RF>1, II=RF.

⑦ QKeras — Quantisation pendant l'entraînement

Au lieu de quantiser un modèle déjà entraîné (post-training quantization), QKeras intègre la quantisation dans la boucle d'entraînement. Le réseau apprend à être précis malgré les contraintes de bits. Résultat : pour le même nombre de bits, QKeras donne une meilleure accuracy que la quantisation post-training.

2.1.1 — Évolution du Design FPGA : HDL → HLS → HLS4ML

Comprendre pourquoi HLS4ML existe — le contexte historique et technique

Niveau 1 — HDL (Hardware Description Language)

Historiquement, programmer un FPGA signifie écrire du VHDL ou Verilog. Ces langages décrivent le matériel au niveau RTL (Register Transfer Level) : tu décris exactement quels registres échangent quelles données à chaque coup d'horloge. Niveau 2 — HLS (High Level Synthesis)

HLS permet d'écrire le comportement du circuit en C ou C++, puis de laisser l'outil (Vivado HLS ou Vitis HLS chez Xilinx/AMD) générer le RTL automatiquement. Tu contrôles l'implémentation via des pragmas HLS insérés dans le code :

// Exemple de pragma HLS dans du code C++
void dense_layer(float inputs[], float outputs[]) {
  #pragma HLS PIPELINE II=1    // pipeline complet
  #pragma HLS UNROLL factor=4  // dérouler 4× la boucle
  for(int i = 0; i < N; i++) {
    outputs[i] = activation(dot_product(weights[i], inputs));
  }
}

Niveau 3 — HLS4ML (Python → HLS → RTL → Bitstream)

HLS4ML ajoute une couche supplémentaire : il prend directement un modèle Keras/PyTorch et génère le code C++ HLS correspondant, avec les types ap_fixed, les pragmas de pipeline, et les connexions entre couches — tout automatiquement. Le développeur contrôle seulement la configuration (précision, ReuseFactor, granularité).

①

Keras Model → hls4ml parse les couches, poids, et activations

②

Code C++ HLS généré → avec ap_fixed, #pragma HLS PIPELINE, réorganisation des couches (ex: fusion BatchNorm)

③

Vivado/Vitis HLS → synthèse du C++ en RTL (VHDL/Verilog automatique)

④

Vivado Implementation → placement et routage sur le FPGA physique → bitstream .bit

Origine CERN — pourquoi HLS4ML a été créé

Au LHC (Large Hadron Collider) au CERN, les détecteurs produisent des données à des débits de plusieurs Tb/s. Le trigger de niveau 1 doit décider en moins de 25 µs si une collision est intéressante, avant de la stocker. Implémenter ce tri en VHDL pour des réseaux neuronaux prendrait des mois de développement et serait non-réutilisable. HLS4ML permet de changer le modèle de classification en quelques heures plutôt que plusieurs mois. [Référence : Duarte et al., JINST 13, P07027 (2018)]

2.1.2 — Architecture interne du FPGA

Comprendre ce que sont réellement les LUTs, DSPs, FFs et BRAMs

La structure fondamentale

Un FPGA est une grille de blocs logiques programmables interconnectés par un réseau de câbles configurables. Contrairement à un CPU qui exécute des instructions séquentiellement, le FPGA câble physiquement les opérations — il n'y a pas de fetch/decode/execute. Tout est du matériel parallèle.

LUT — Look-Up Table

Le bloc de base du FPGA. Une LUT à 6 entrées est essentiellement une petite ROM de 64 bits : pour chaque combinaison de 6 inputs, elle retourne 1 bit de sortie. En programmant cette ROM, on peut implémenter n'importe quelle fonction logique à 6 variables. Les additions, comparaisons, et même des portions d'activations sont implementées en LUTs. Sur le Zynq-7020 : 53 200 LUTs.

DSP48E1 — Digital Signal Processor

Un bloc DSP est un multiplieur-accumulateur hardware optimisé. Il fait P = A×B + C en un seul cycle d'horloge, avec A sur 25 bits et B sur 18 bits. Une multiplication 16-bit × 16-bit tient dans 1 seul DSP car 16 < 25 et 16 < 18. C'est le bloc utilisé pour les multiplications des poids dans les couches Dense. Sur le Zynq-7020 : 220 DSPs = 220 multiplications 16-bit simultanées par cycle d'horloge — la contrainte principale de tout le tutoriel.

Flip-Flop (FF) — Registre

Un FF stocke 1 bit d'état, synchronisé sur l'horloge. Les FFs sont utilisés comme registres de pipeline : ils mémorisent les résultats intermédiaires entre étages de pipeline. Sur le Zynq-7020 : 106 400 FFs (= 53 200 paires de LUT-FF dans les slices).

Block-RAM (BRAM)

Des blocs de mémoire SRAM configurables (36 Kb chacun). Utilisés pour stocker les poids volumineux ou les buffers d'activation qui ne rentrent pas dans les LUTs. Sur le Zynq-7020 : 140 BRAMs de 36 Kb = 5 Mb total.

Pourquoi les DSPs sont la contrainte critique

Un multiplieur 16×16 bits (ap_fixed<16,6> × ap_fixed<16,6>) utilise typiquement 3 DSPs. Un réseau avec 1000 connexions en RF=1 a besoin de 3000 DSPs — 13× la capacité du PYNQ-Z1. En réduisant à 6 bits (ap_fixed<6,2> × ap_fixed<6,2>), le multiplieur tient dans 1 LUT + 0 DSP. C'est pourquoi QKeras 6-bit est si efficace.

Zynq-7020 — Architecture SoC

Le PYNQ-Z1 utilise un Zynq-7020 qui est un SoC (System on Chip) : il combine un processeur ARM Cortex-A9 dual-core (PS — Processing System) avec la logique FPGA programmable (PL — Programmable Logic). HLS4ML déploie le réseau neuronal dans la PL, et Python sur l'ARM contrôle le flux de données via DMA (Part 7).

2.2.1 — Parallélisme et Pipeline

Les deux stratégies d'accélération fondamentales que HLS4ML exploite

① Parallélisme — plusieurs opérations simultanées

Sur CPU : les couches d'un réseau neuronal s'exécutent séquentiellement — fc1 finit, puis fc2 commence. Sur FPGA avec HLS4ML : toutes les couches peuvent avoir leur propre circuit hardware câblé en parallèle. Pendant que fc2 traite la sortie de fc1, fc1 peut déjà commencer à traiter le sample suivant.

Analogie : chaîne de montage vs atelier artisanal

Un atelier artisanal (CPU) : un ouvrier fait tout le travail de A à Z sur une pièce. Productivité limitée par la vitesse d'un ouvrier.
Une chaîne de montage (FPGA) : chaque poste fait une seule opération, mais plusieurs pièces avancent simultanément dans la chaîne. Productivité = débit de la chaîne entière.

② Pipelining — chevauchement des traitements

Le pipelining signifie que pendant qu'une couche traite le sample N, la couche suivante traite déjà le sample N-1. En HLS4ML, le pragma #pragma HLS PIPELINE II=1 demande au compilateur d'organiser le circuit pour accepter 1 nouveau sample par cycle d'horloge.

Latency vs Throughput — deux métriques différentes

Latency (latence)

Temps entre l'entrée d'UN sample et la sortie de son résultat. Dans ton modèle Part 1 : 50 cycles = 0.250 µs. Critique pour les applications temps-réel strictes (trigger LHC, systèmes de contrôle).

Throughput (débit)

Combien de samples peuvent être traités par unité de temps. Avec II=1 : 1 sample/cycle = 200 millions d'inférences/seconde à 200MHz. Critique pour les applications à haut débit (classification continue de flux).

L'interaction avec le ReuseFactor

Avec RF=1, le circuit est entièrement parallèle et pipeliné → II=1. Avec RF>1, certains calculs sont séquentiels (un multiplicateur partagé entre plusieurs poids) → II = RF. C'est le compromis fondamental du ReuseFactor.

Résumé Part 1 — Parallelism vs Ressources

Part 1 avec RF=1 : pipelining parfait (II=1), latence minimale (50 cycles), mais 2198 DSPs requis pour 220 disponibles. C'est le prix du parallélisme total — chaque multiplication a son propre hardware câblé physiquement.

2.2.2 — ReuseFactor en détail

Le levier principal pour ajuster le compromis latence ↔ ressources

Définition précise

Le ReuseFactor (RF) détermine combien de fois une unité de calcul physique (un multiplicateur DSP) est réutilisée pour calculer différents produits poids × activation. Il répond à la question : "Est-ce que je câble un multiplicateur par poids (RF=1) ou est-ce que je fais passer plusieurs poids à travers le même multiplicateur séquentiellement (RF>1) ?"

DSPs_utilisés = ⌈ N_multiplications ÷ ReuseFactor ⌉

Stratégie mixte par couche — pourquoi c'est optimal

La première couche (fc1) a généralement beaucoup plus de poids que la dernière (output). Donner le même RF à toutes les couches est sous-optimal : la petite couche de sortie n'a pas besoin de RF élevé. La stratégie mixte donne un RF élevé aux grandes couches et un RF faible aux petites.

# Exemple de stratégie mixte pour ton réseau
config['LayerName']['fc1']['ReuseFactor']    = 8   # grande couche → partager les DSPs
config['LayerName']['fc2']['ReuseFactor']    = 4   # couche intermédiaire
config['LayerName']['fc3']['ReuseFactor']    = 2   # couche plus petite
config['LayerName']['output']['ReuseFactor'] = 1   # petite couche → garder II=1 ici
Règle empirique — 80% de capacité DSP maximum

Ne jamais utiliser plus de 80% des DSPs disponibles. Les 20% restants sont réservés pour : la logique de contrôle (AXI interface, DMA), les marges de routage, et les circuits d'horloge. Un FPGA chargé à 95% de DSPs devient souvent impossible à router correctement par Vivado.

Pour PYNQ-Z1 : maximum 176 DSPs (80% × 220). En Part 7, tu en utilises seulement 9 — très confortable.

2.2.3 — Precision : ap_fixed — Festkommadarstellung

La représentation des nombres en virgule fixe — formules exactes, diagramme bit par bit, exemples

Pourquoi pas float32 sur FPGA ?

Float32 a 1 bit signe + 8 bits exposant + 23 bits mantisse = 32 bits → un multiplieur 32×32 bits qui consomme environ 14-16 DSPs par multiplication. Avec ap_fixed<8,1>, le multiplieur tient dans 1 DSP ou même 0 DSP (LUTs seules). C'est pour ça que la quantisation est indispensable sur FPGA.

Nomenclature — ap_fixed<W, I>

W — Word length (Gesamtbitbreite)

Nombre total de bits. C'est le premier paramètre. Plus W est grand, plus la représentation est précise, mais plus on consomme de ressources FPGA.

I — Integer bits (Vorkommabits)

Nombre de bits réservés à la partie entière (signe inclus). Le signe occupe 1 bit sur les I. Les W−I bits restants sont pour la partie fractionnaire.

❌ Overflow — I trop petit (le pire)

# 5.3 → wrap-around → -2.7 !!

La résolution est trop grossière → des valeurs proches sont arrondies à la même valeur. Le réseau perd de la précision fine mais ne produit pas de résultats catastrophiques. Souvent acceptable si la perte d'accuracy reste sous 1-2%.

2.2.5 — Pruning et Sparsité

Mettre des poids à zéro pendant l'entraînement pour économiser les DSPs

Principe fondamental

Un réseau neuronal dense de base utilise chaque connexion (poids) avec une valeur non-nulle. Le pruning consiste à mettre délibérément certains poids à zéro. Un poids nul × une activation = 0 → multiplication inutile → DSP économisé.

Pourquoi ça marche ? — La surparamétrisation des réseaux

Les réseaux neuronaux typiques sont massivement surparamétrés : ils ont beaucoup plus de poids qu'il n'en faut pour la tâche. Des études (Lottery Ticket Hypothesis, Frankle et al. 2018) montrent qu'on peut souvent réduire 50-90% des poids à zéro sans perte d'accuracy significative, à condition de le faire progressivement pendant l'entraînement.

Types de Pruning

Magnitude Pruning (utilisé en Part 3 & 4)

Les poids avec la plus petite valeur absolue sont mis à zéro. Intuition : un poids proche de zéro a peu d'influence sur la sortie. TensorFlow Model Optimization Toolkit fait ça progressivement :

Entraînement normal → début pruning (10% → 30% → 50%) → fine-tuning final.

Structured vs Unstructured Pruning

Unstructured : des poids individuels aléatoires → sparsité irrégulière → difficile à exploiter en hardware.
Structured : des neurones entiers, des filtres entiers → blocs de zéros réguliers → plus efficace sur FPGA. HLS4ML recommande le structured pruning.

Comment HLS4ML exploite la sparsité

HLS4ML peut exploiter les zéros seulement si le ReuseFactor est un multiple du pattern de sparsité. Si 50% des poids sont zéro et RF=2, alors chaque multiplicateur hardware ne voit en moyenne qu'un seul poids non-nul par paire → son usage est divisé par 2.

# Activer l'option Unrolled pour exploiter la sparsité
config['LayerName']['fc1']['Unrolled'] = True
# → hls4ml génère du code HLS qui saute les multiplications par zéro

Activations & Batch Normalization

Pourquoi les activations n'utilisent quasiment pas de DSPs — et comment HLS4ML fait disparaître la BatchNorm

① À quoi sert une fonction d'activation ?

Sans fonction d'activation, un réseau Dense de N couches est mathématiquement équivalent à une seule couche Dense. La composition de transformations linéaires reste linéaire — le réseau ne peut apprendre que des frontières de décision linéaires, quel que soit le nombre de couches.

Preuve — 2 couches Dense sans activation = 1 seule couche
couche1 : y₁ = W₁ · x + b₁  ← linéaire
couche2 : y₂ = W₂ · y₁ + b₂  ← linéaire
→ y₂ = W₂·(W₁·x + b₁) + b₂ = (W₂·W₁)·x + (W₂·b₁ + b₂)
→ y₂ = W_eq·x + b_eq    ← équivalent à 1 seule couche !

ReLU casse cette linéarité : max(0, y₁) n'est pas linéaire → les couches successives peuvent apprendre des représentations non-linéaires complexes.

② Les activations dans ton réseau — DSPs et LUTs

softmax Softmax exp(xᵢ) / Σexp(xⱼ) Sortie → probabilités (somme = 1) ~0 Modéré (LUT-tables pour exp())

③ Pourquoi ReLU ne coûte rien sur FPGA

Sur CPU

if (x > 0) return x; else return 0;
Un test conditionnel. Simple mais séquentiel.

Sur FPGA — câblage direct

Juste regarder le bit de signe (MSB) :
signe = 0 → x ≥ 0 → sortie = x
signe = 1 → x < 0 → sortie = 0
1 LUT suffit. 0 DSP. Latence = 1 cycle.

Cela signifie que pour le type de résultat de ces couches, I=1 couvre presque toujours la plage (valeurs ≥ 0).

④ Batch Normalization — qu'est-ce que c'est ?

La BatchNorm normalise les sorties de chaque couche pendant l'entraînement pour stabiliser et accélérer la convergence. Elle applique à chaque activation x :

y = γ × (x − μ) / √(σ² + ε) + β
ParamètreTypeSignificationValeur à l'inférence
μStatistiqueMoyenne des activations (calculée sur le batch d'entraînement)Fixe — moving average
σ²StatistiqueVariance des activationsFixe — moving average
γApprisFacteur d'échelle — le réseau choisit l'amplitudePoids appris, fixe
βApprisDécalage — le réseau choisit le centrePoids appris, fixe
εConstantePetit nombre pour éviter la division par 0≈ 1e-5

⑤ Simplification à l'inférence → opération affine

À l'inférence, μ et σ² sont figés. La formule se réduit à une simple multiplication + addition :

y = γ × (x − μ) / √(σ² + ε) + β
y = x × [ γ / √(σ² + ε) ] + [ β − μ × γ / √(σ² + ε) ]
y = x × a + b     ← constantes calculables avant synthèse FPGA

⑥ La fusion BatchNorm dans HLS4ML

HLS4ML fusionne automatiquement la BatchNorm dans la couche Dense précédente. Les poids et biais de la Dense sont modifiés pour absorber a et b :

Avant fusion :
  Dense : z = W · x + b_d
  BatchNorm : y = a × z + b_bn
Après fusion (équivalent exact) :
  Dense fusionnée : y = (a·W) · x + (a·b_d + b_bn)
✅ BatchNorm = 0 ressource supplémentaire sur FPGA

Après fusion, la BatchNorm disparaît du modèle HLS. Ses paramètres sont absorbés dans les poids et biais Dense. Résultat : 0 DSP et 0 LUT supplémentaires. C'est gratuit — c'est l'une des raisons pour lesquelles HLS4ML recommande d'utiliser BatchNorm pendant l'entraînement sans crainte pour l'inférence FPGA.

⑦ Pourquoi les graphiques ① et ② diffèrent quand il y a des BatchNorm

① Poids — AVANT optimisation

Montre fc1, bn1, fc2, bn2, fc3, bn3 séparément. Les couches bn ont leurs propres distributions de γ et β bien distinctes.

② Poids — APRÈS optimisation ← à utiliser

Les couches bn1, bn2, bn3 ont disparu. Les poids fc1, fc2, fc3 ont changé (multipliés par a). Les distributions sont différentes → il faut utiliser ② pour choisir ap_fixed.

Ton cas — pas de BatchNorm → ① ≈ ②

Ton réseau fc1→relu→fc2→relu→fc3→relu→output→softmax n'a pas de BatchNorm. HLS4ML n'a rien à fusionner → les distributions avant et après sont quasi-identiques. C'est normal et attendu.

2.2.6 — Latency et Initiation Interval (II)

Deux métriques distinctes qui mesurent des aspects différents de la performance temporelle

Définitions précises

Latency (Latenz)

Nombre de cycles d'horloge nécessaires pour qu'un sample entre dans le réseau et que son résultat sorte. Mesure le temps de réponse pour un seul sample.

Latence [µs] = Cycles ÷ Fréquence_horloge [MHz]

PYNQ-Z1 @ 100 MHz : 50 cycles = 0.500 µs
PYNQ-Z1 @ 200 MHz : 50 cycles = 0.250 µs

Initiation Interval (II)

Nombre de cycles d'horloge avant que le système soit prêt à accepter un nouveau sample. Mesure le débit maximum.

Débit [samples/s] = Fréquence [Hz] ÷ II

II=1 → 200M samples/s à 200MHz
II=94 → ~2.1M samples/s (Part 7)

Relation avec le ReuseFactor

II = ReuseFactor (dans le cas général avec pipelining HLS)

Avec RF=1 : chaque multiplication a son propre hardware → II=1 (pipelining complet). Avec RF=64 : 64 poids partagent un multiplieur → il faut 64 cycles pour traiter tous les poids d'une couche → II=64 (approximativement, dépend de l'architecture exacte).

Pourquoi la Part 6 (CNN) a II=1028 ?

Le CNN utilise io_type='io_stream' : les pixels d'une image 32×32 arrivent en séquence (1 pixel par cycle). Pour traiter une image complète, il faut 32×32 = 1024 cycles d'entrée + quelques cycles de finalisation = II≈1028. Ce n'est pas lié au ReuseFactor mais à la nature séquentielle du streaming de données d'image.

io_parallel vs io_stream — le choix architectural

io_parallel (défaut pour MLP)

Toutes les entrées arrivent simultanément en un seul cycle. Latence minimale, mais consomme beaucoup de mémoire si les vecteurs d'entrée sont grands. Utilisé en Parts 1-5.

io_stream (pour CNN et grandes entrées)

Les données arrivent en séquence (streaming). Réduit la mémoire de tampon nécessaire. II élevé mais mémoire minimale. Nécessaire pour les images CNN qui ont des milliers de valeurs d'entrée.

Quand utiliser 'model'

Pour les premiers tests ou quand toutes les couches ont des distributions similaires. Inconvénient : si fc1 a besoin de ap_fixed<8,1> mais que la couche output a besoin de ap_fixed<12,3>, on est bloqué à la précision la plus haute — gaspillage.

② Niveau par type — granularity='type'

Permet de donner des précisions différentes selon le type de couche (Dense, Activation, BatchNorm, etc.). Utile pour distinguer les couches de calcul des couches d'activation.

# Configuration par type de couche
config = hls4ml.utils.config_from_keras_model(model, granularity='type')

# Précision pour tous les Dense
config['LayerType']['Dense']['Precision']      = 'ap_fixed<10,2>'
config['LayerType']['Dense']['ReuseFactor']   = 4

# Précision pour toutes les Activation (ReLU, etc.)
config['LayerType']['Activation']['Precision'] = 'ap_fixed<12,3>'

# Priorité : LayerName > LayerType > Model
# Si une couche a une config 'name', elle écrase 'type'

③ Niveau par couche — granularity='name' ← recommandé

Le niveau le plus puissant. Chaque couche, identifiée par son nom, reçoit sa propre configuration. On peut contrôler séparément les poids, les biais, et les résultats (sorties) de chaque couche.

# Configuration par couche — Part 2 et Part 4
config = hls4ml.utils.config_from_keras_model(model, granularity='name')

# ── Activer le Trace sur toutes les couches (pour débogage) ──
for layer in config['LayerName'].keys():
    config['LayerName'][layer]['Trace'] = True

# ── Précision des poids par couche ──
# Cap droit poids ≈ 2⁻¹ → I = 0+1 = 1 → ap_fixed<8,1>
config['LayerName']['fc1']['Precision']['weight']   = 'ap_fixed<8,1>'
config['LayerName']['fc2']['Precision']['weight']   = 'ap_fixed<8,1>'
config['LayerName']['fc3']['Precision']['weight']   = 'ap_fixed<8,1>'
config['LayerName']['output']['Precision']['weight'] = 'ap_fixed<8,1>'

# ── Précision des biais (plus concentrés → moins de bits) ──
for layer in ['fc1', 'fc2', 'fc3', 'output']:
    config['LayerName'][layer]['Precision']['bias'] = 'ap_fixed<10,2>'

# ── Précision des résultats (sorties des couches) ──
# Cap droit activations ≈ 2² → I = 2+1 = 3 → ap_fixed<12,3>
for layer in ['fc1', 'fc2', 'fc3']:

# ── ReuseFactor mixte par couche ──

config['LayerName']['fc2']['ReuseFactor']    = 2
config['LayerName']['fc3']['ReuseFactor']    = 1   # petite couche

Ce que hls4ml calcule automatiquement avec granularity='name'

Les accumulateurs internes de chaque couche Dense sont calculés automatiquement. Un accumulateur stocke la somme partielle des produits (poids × entrée) pendant qu'on parcourt tous les neurones. Sa taille doit éviter l'overflow. hls4ml calcule cette taille sur le pire cas théorique (toutes entrées à leur max théorique), ce qui est souvent 2-4× trop large par rapport aux valeurs réelles. Le profiling permet de le réduire manuellement.

Hiérarchie de priorité des configurations

LayerName > LayerType > Model (la plus spécifique l'emporte)
# Exemple : on peut mixer les niveaux
config = hls4ml.utils.config_from_keras_model(model, granularity='name')

# Écrase pour tous les Dense

# Écrase pour la couche fc1 spécifiquement (priorité max)

④ HGQ — High Granularity Quantization (Part 4.1)

HGQ va au-delà du niveau 'name' : il assigne une précision à chaque poids individuel, basée sur son importance pour le résultat final. Les poids importants gardent plus de bits, les poids marginaux en reçoivent moins.

1

Entraînement HGQ : le réseau apprend simultanément les valeurs de poids ET leur précision optimale. Objectif : minimiser les BOPs (Bit Operations) = mesure de l'utilisation hardware totale.

# Cible : 6500 BOPs ≈ 6500 LUTs équivalents
# (DSPs comptent pour plus de LUTs — pondérés)
2

Proxy Model obligatoire : HLS4ML ne comprend pas HGQ directement. Il faut créer un Proxy Model — un émulateur bit-exact du modèle HGQ — avec des types ap_fixed fixes dérivés des précisions HGQ.

from hgq.proxy import to_proxy_model
proxy = to_proxy_model(hgq_model, cover_range=1.0)
# cover_range : facteur de sécurité
# 1.0 = couvre 100% des valeurs vues pendant trace_minmax
# Trop bas → overflow silencieux → accuracy effondrée (ton cas Part 4.1)
3

Conversion hls4ml du Proxy : le Proxy se comporte comme un modèle Keras normal → hls4ml peut le convertir directement.

hls_model = hls4ml.converters.convert_from_keras_model(
    proxy, output_dir='model_hgq',
    part='xc7z020clg400-1', backend='Vitis'
)
Pourquoi Part 4.1 échoue — accuracy 46.7%

Le cover_range=1.0 était insuffisant : certaines valeurs vues pendant l'inférence réelle dépassaient la plage couverte pendant le trace_minmax → overflow silencieux → ~30% des samples produisent des résultats faux → accuracy 46.7% au lieu de 75.9%. Solution : augmenter le cover_range (ex: 1.2 ou 1.5) ou réduire la cible BOPs.

① Xilinx toolchain native Linux : Vivado et Vitis HLS sont développés et optimisés pour Linux. Sous Windows, beaucoup de scripts d'automatisation ne fonctionnent pas ou génèrent des erreurs de chemins.

② Variables d'environnement : HLS4ML doit appeler Vivado/Vitis directement depuis Python. Il utilise le source de Bash pour charger les variables :

source /opt/Xilinx/Vivado/2022.1/settings64.sh

③ Chemins de fichiers : Vivado génère des centaines de fichiers dans des sous-dossiers profonds. Sous Windows, les chemins dépassent souvent la limite MAX_PATH (260 caractères) → erreurs de synthèse difficiles à diagnostiquer.

④ Packages Python natifs : Certaines dépendances (uproot, awkward-array pour les données HEP) ont des roues binaires Linux uniquement. Conda gère ça proprement sous Linux.

Multiplications par couche & DSPs

La seule formule à retenir — et sa vérification sur chaque Part du PDF

① Pourquoi une couche Dense fait des multiplications

Une couche Dense implémente un produit matrice-vecteur. Chaque neurone de sortie j calcule la somme de tous les produits poids × entrée :

sortie[j] = Σᵢ ( poids[j][i] × entrée[i] ) + biais[j]

Chaque terme poids[j][i] × entrée[i] est 1 multiplication → utilise un DSP. L'addition + biais[j] est une simple addition → utilise des LUTs, pas de DSP.

Exemple minimal — Dense(3 → 2) : 3 entrées, 2 neurones de sortie
sortie[0] = w[0][0]×x₀ + w[0][1]×x₁ + w[0][2]×x₂ + b[0]  ← 3 multiplications
sortie[1] = w[1][0]×x₀ + w[1][1]×x₁ + w[1][2]×x₂ + b[1]  ← 3 multiplications
Total = 2 sorties × 3 entrées = 6 multiplications = N_sorties × N_entrées

② La formule exacte du nombre de multiplications

N_muls (couche Dense) = N_entrées × N_sorties

C'est le nombre d'éléments dans la matrice de poids. Chaque élément fait exactement 1 multiplication. C'est exact, pas une approximation.

Couches sans multiplication (0 DSP)

ReLU / relu : max(0, x) → comparaison simple → LUTs, 0 DSP
BatchNorm : HLS4ML la fusionne dans la Dense précédente → 0 DSP supplémentaire
Softmax : exp() → implémenté via LUT-tables → quasiment 0 DSP
→ Seules les couches Dense et Conv comptent pour les DSPs.

④ De N_muls aux DSPs — la formule complète

Formule approchée (les 3 facteurs)
DSPs ≈ ⌈ N_muls × (1 − s) × α(W) ÷ RF ⌉ + β
N_muls = Σ (N_entrées × N_sorties) sur toutes les couches Dense  [exact]
s = fraction de poids mis à zéro par pruning  [exact]
α(W) = DSPs par multiplication selon W bits  [empirique Vivado]
RF = ReuseFactor  [exact diviseur]
β = overhead HLS (accumulateurs + AXI + contrôle)  [variable]

α(W) — DSPs par multiplication selon la précision

Le DSP48E1 du Zynq-7020 fait P = A×B + C avec A sur 25 bits et B sur 18 bits. Une multiplication 16-bit × 16-bit (W=16) tient donc dans 1 seul DSP (16 < 25 et 16 < 18). La valeur α est empirique — déduite des résultats de synthèse Vivado du PDF, pas calculée analytiquement.

Pourquoi α(16) ≈ 0.52 et pas 1.0 si 1 mul = 1 DSP ?

Théoriquement, 1 multiplication 16-bit = 1 DSP48E1. Pour 4256 muls en RF=1, on attendrait 4256 DSPs. Mais le PDF dit 2198 — soit α ≈ 0.52. Pourquoi ?

Le DSP48E1 fait P = A×B + C : la multiplication ET l'accumulation dans le même bloc. Cela signifie que chaque DSP fait à la fois le produit poids×entrée ET ajoute le résultat à la somme partielle — deux opérations pour le prix d'une. Vivado/Vitis HLS exploite ce mécanisme en cascade : 1 DSP ≈ 2 multiplications pipelinées, d'où α ≈ 0.52 ≈ 1/2.

Ces accumulateurs doivent être assez larges pour ne pas déborder → ils utilisent des DSPs supplémentaires au-delà des multiplications elles-mêmes. β diminue quand RF augmente (moins de circuits parallèles = moins d'accumulateurs).

Résumé — comment réduire les DSPs

1. Réduire N_muls → changer l'architecture (moins de neurones) → impact sur accuracy, difficile
2. Augmenter s (pruning) → multiplier par (1−s) → Part 3 : ×0.5, Part 4 : ×0.25
3. Réduire α via W (précision) → QKeras 6-bit : α = 0.05 au lieu de 0.52 → ÷10
4. Augmenter RF → diviser par RF → Part 7 : ÷64

La combinaison 2+3 (Part 4) est la plus efficace : 4256 × 0.25 × 0.05 ÷ 1 = 53 muls-DSPs seulement. C'est QKeras + pruning qui "casse" le problème, pas le RF seul.

La ligne horizontale + les 2 petits traits verticaux aux extrémités.

Le rectangle bleu foncé.

⚠ LA RÈGLE FONDAMENTALE

La zone grise DOIT couvrir le cap DROIT du segment. Si le cap droit dépasse la zone grise → overflow/saturation → le réseau produit des résultats faux même si l'accuracy Keras était parfaite. Le cap gauche non couvert = perte de précision fine, souvent acceptable.

Comment lire le profiling

4 questions à poser dans l'ordre, pour chaque ligne du graphique

1

Lire le cap droit du segment — trouver "a" sur l'axe X

L'axe X est en puissances de 2 : 2⁻¹⁹, 2⁻¹⁵, …, 2⁻¹, 2⁰, 2¹, etc. Le cap droit (moustache droite) se trouve à 2a pour une certaine valeur de a. Cette valeur est la magnitude maximale de tes poids/activations.

I nécessaire = a + 1

Explication : pour représenter 2a sans overflow, on a besoin que 2(I−1) ≥ 2a, donc I−1 ≥ a, donc I ≥ a+1.

Part 4.1 — HGQ (High Granularity Quantization)

Quantisation par poids individuel — résultat mitigé

Qu'est-ce que HGQ ?

HGQ (High Granularity Quantization) va encore plus loin que QKeras : au lieu d'attribuer la même précision à tous les poids d'une couche, il détermine la précision idéale pour chaque poids individuel. Les poids importants gardent plus de bits, les poids moins importants en reçoivent moins.

Workflow spécifique HGQ

1

Entraînement HGQ — assignation automatique de bits par poids

Objectif : 6500 BOPs (Bit Operations) ≈ 6500 LUTs équivalents (incluant DSPs pondérés).

2

Conversion en Proxy Model — étape obligatoire

HLS4ML ne peut pas convertir directement un modèle HGQ. Il faut d'abord créer un Proxy Model avec trace_minmax et un cover_range (facteur de sécurité). Ce proxy est un émulateur bit-exact du modèle HGQ.

3

Conversion hls4ml du Proxy

hls_model = hls4ml.converters.convert_from_keras_model(
    proxy, output_dir='model_3.1/hls4ml_prj',
    part='xc7z020clg400-1', backend='Vitis'
)

Part 5 — BDT / Conifer

Sortie des réseaux neuronaux — arbres de décision boostés sur FPGA

Qu'est-ce qu'un BDT (Boosted Decision Tree) ?

Un BDT est un modèle ML basé sur des arbres de décision, entraîné avec XGBoost (eXtreme Gradient Boosting). Chaque arbre corrige les erreurs du précédent. Pour les FPGAs, le framework Conifer convertit ces modèles en HLS, avec plusieurs backends : Vivado HLS, VHDL, et Julia.

Workflow BDT → FPGA

import xgboost as xgb
import conifer

# Entraîner le BDT avec XGBoost
clf = xgb.XGBClassifier(n_estimators=20, max_depth=3)
clf.fit(X_train, y_train)

# Convertir avec Conifer
cfg = conifer.backends.xilinxhls.auto_config()
cfg['OutputDir'] = 'conifer_prj'
model = conifer.model(clf, conifer.backends.xilinxhls, cfg)
model.compile()  # compile et synthétise

# Déploiement sur PYNQ-Z1
accelerator = conifer.backends.xilinxhls.runtime.ZynqDriver(
    'conifer_jettag.bit', batch_size=1

Part 6 — CNNs (Convolution sur FPGAs)

Dataset SVHN (images 32×32 RGB) — les CNNs sont bien plus gourmands que les MLP

Pourquoi les CNNs sont plus difficiles ?

Un réseau dense (MLP) fait des multiplications entre vecteurs. Un CNN fait des convolutions 2D — des milliers de multiplications pour chaque patch de l'image. Sur le dataset SVHN (Street View House Numbers, images 32×32 RGB), le CNN pruné + quantisé a encore 5386 DSPs en configuration parallèle.

io_type='io_stream' — spécifique aux CNNs

Pour limiter la mémoire nécessaire au traitement des images (32×32 pixels × 3 canaux = 3072 valeurs), on utilise le mode streaming (io_type='io_stream'). Backend VivadoAccelerator — ce qui change

Part 8 — Symbolic Regression

Remplacer un réseau neuronal par une équation mathématique compacte

Principe — une approche radicalement différente

Au lieu d'un réseau de neurones, on cherche une expression mathématique qui décrit les données. Normalement on utilise PySR (genetic programming) pour trouver automatiquement ces expressions. PySR n'ayant pas pu être installé, les expressions ont été définies manuellement et parsées avec Sympy.

Deux implémentations comparées

Standard HLS

Utilise les bibliothèques math standard de HLS pour calculer les expressions (sin, exp, log, etc.). Précision maximale mais consomme plus de ressources.

HLS avec Look-Up Tables (LUT)

Les fonctions complexes sont approximées par des tables précalculées. Moins précis mais beaucoup plus rapide et moins gourmand en ressources.

La leçon principale

La quantisation seule (réduire les bits) est plus efficace que le pruning seul pour réduire les DSPs. La combinaison QKeras (6-bit pendant entraînement) + pruning (75% zéros) est la clé du succès en Part 4. Le ReuseFactor = 64 en Part 7 réduit encore les DSPs de 122 → 9 au prix d'un II=94, ce qui reste acceptable pour cette application.

Problèmes rencontrés

Les 3 obstacles principaux lors de la mise en œuvre

① Compatibilité système

HLS4ML requiert une combinaison très spécifique de versions : Python, TensorFlow, QKeras, hls4ml, et Vivado/Vitis HLS doivent toutes être compatibles entre elles. Un seul désaccord de version peut bloquer l'ensemble de la chaîne.

Solution adoptée

Ubuntu 22.04 LTS (Linux natif pour Xilinx). Conda pour isoler l'environnement. Fichier environment.yml fourni par le tutoriel officiel. Ajout manuel de conifer (pour BDT) et Julia (pour Conifer-Julia backend). La toolchain Xilinx est sourcée via source /opt/Xilinx/Vivado/20xx.x/settings64.sh dans le terminal Bash.

② Conflit Vitis Backend — Problème de version

La version de hls4ml utilisée appelait le programme vitis_run pour lancer la synthèse HLS. Or, Vitis 2022 utilise l'exécutable vitis_hls. De plus, d'anciennes conventions dans hls4ml attendaient encore vivado_hls. Résultat : erreurs systématiques lors du build automatique.

Symptôme

Les builds échouaient avec "command not found: vitis_run" ou des chemins incorrects vers les exécutables HLS. Le projet HLS était généré correctement mais la synthèse ne se lançait pas.

Solution adoptée

Downgrade de hls4ml vers une version plus stable compatible avec Vitis 2022. La compatibilité entre versions de hls4ml et Vivado/Vitis est documentée dans la matrice de compatibilité du framework — à vérifier avant l'installation.

③ PYNQ-Z1 vs PYNQ-Z2 — Board non officiel

HLS4ML et Conifer supportent officiellement le PYNQ-Z2, pas le PYNQ-Z1. Bien que les deux utilisent le même chip Zynq xc7z020clg400-1, les Board Definition Files (BDF) sont différents — ils définissent les contraintes de placement, les ports physiques, et les périphériques.

Solution adoptée

Ajout manuel des fichiers Board Definition du PYNQ-Z1 dans les répertoires système des frameworks (Vivado et Conifer). Source : https://discuss.pynq.io/t/addition-of-pynq-z1-board-into-vivado/3556. Une fois les BDF ajoutés, le chip part='xc7z020clg400-1' est correctement reconnu pour la synthèse et l'implémentation.
